Módulo 1 · FastAPI

main.py: el portero

Cuando llega cualquier petición HTTP a localhost:8000, el primero en verla es FastAPI. Actúa como un recepcionista que decide quién atiende cada petición.

from fastapi import FastAPI
from routers import boards, analytics

app = FastAPI(title="ChanDB API")

app.add_middleware(CORSMiddleware,
  allow_origins=["http://localhost:5173"])

app.include_router(boards.router)
app.include_router(analytics.router)
→ Importa FastAPI
→ Importa los módulos de rutas
 
→ Crea la app — el punto de entrada
 
→ Permite peticiones desde el frontend
 
→ Registra el router de boards
→ Registra el router de analytics

FastAPI lee estas líneas al arrancar y construye un mapa interno: "si llega GET /api/boards, llama a esta función". Los routers son como departamentos dentro del recepcionista.

Módulo 1 · database.py

database.py: el mensajero

Todos los routers llaman a la misma función q(). Es el único punto que habla con Neo4j.

# database.py — importado en todo el backend
async def run_query(query, **params):
  async with driver.session() as session:
    result = await session.run(query, **params)
    raw = await result.data()
    return [serialize(row) for row in raw]
→ Comentario: todos los routers usan esto
→ Define función asíncrona que recibe un query
→ Abre conexión a Neo4j (la cierra al terminar)
→ Envía el Cypher y espera que empiece
→ Recoge todos los resultados como lista
→ Convierte objetos Neo4j a JSON y devuelve
💡 ¿Por qué async/await?

Async significa que el servidor no se congela mientras espera a Neo4j. Es como un camarero que atiende otras mesas mientras tu pedido está en cocina, en lugar de quedarse parado frente a ti.

Módulo 2 · El Grafo

¿Por qué no usamos SQL?

En SQL, todo vive en tablas. Relacionar un Post con su User requiere un JOIN: buscar el id del usuario en otra tabla. En un foro con 4,000 posts y 15 tipos de relación, esos JOINs se acumulan y se vuelven lentos.

🗂 SQL (tablas)

Para saber "¿quién escribió este post?":

SELECT u.alias FROM users u
JOIN posts p ON p.user_id = u.id
WHERE p.id = '...'

Necesitas saber de antemano cómo unir tablas. Si agregas un nuevo tipo de relación, cambias el esquema.

🔵 Neo4j (grafo)

Para saber "¿quién escribió este post?":

MATCH (u:User)-[:WROTE]->(p:Post)
WHERE p.id = '...'
RETURN u.alias

La relación WROTE ya existe en el grafo. Seguirla es tan rápido como cruzar un puente directo.

Módulo 3 · Cypher

Cypher: hablarle al grafo

Cypher es el lenguaje de consultas de Neo4j. En vez de decir "busca en la tabla posts donde user_id = X", describes un patrón visual: "búscame un User conectado con WROTE a un Post".

Las 4 operaciones principales

MATCH — Buscar patrones
CREATE — Crear nodos / relaciones
SET / REMOVE — Modificar propiedades
DETACH DELETE — Eliminar con relaciones

La sintaxis visual clave

(n:Label) → nodo con label
-[:TIPO]-> → relación con dirección
n.propiedad → acceder propiedad
$param → parámetro seguro
Módulo 3 · MATCH y CREATE

Buscar y crear en el grafo

Aquí están los dos Cypher más frecuentes del proyecto, con cada línea traducida.

// Buscar todos los boards
MATCH (b:Board)
RETURN b
ORDER BY b.name
→ Comentario en Cypher
→ Encuentra todos los nodos de tipo Board
→ Devuélvelos como resultado
→ Ordenados por nombre alfabéticamente
// Crear un nuevo Board
CREATE (b:Board {
  id: randomUUID(),
  name: $name,
  slug: $slug,
  created_at: date()
})
RETURN b
→ Comentario
→ Crea un nodo nuevo de tipo Board con propiedades:
→   id generado automáticamente (UUID único)
→   nombre desde el parámetro $name
→   slug desde el parámetro $slug
→   fecha actual al momento de crear
 
→ Devuelve el nodo recién creado
Módulo 3 · SET y DELETE

Modificar y eliminar

// Actualizar karma de usuario
MATCH (u:User {id: $id})
SET u.karma = u.karma + $delta
RETURN u
→ Comentario
→ Encuentra el User cuyo id coincide con $id
→ Incrementa su karma sumando $delta al valor actual
→ Devuelve el nodo actualizado
// Eliminar un Post con todo
MATCH (p:Post {id: $id})
DETACH DELETE p
→ Comentario
→ Encuentra el Post con ese id
→ DETACH borra primero las relaciones; luego borra el nodo
⚠️ Por qué DETACH y no solo DELETE

Neo4j no permite borrar un nodo que tenga relaciones activas — te daría error. DETACH DELETE corta todas las relaciones automáticamente antes de borrar. Si solo quieres quitar una propiedad, usas REMOVE n.propiedad.

Módulo 4 · nodes.py

SET propiedad en cualquier nodo

nodes.py es el router más genérico: opera sobre cualquier label autorizado. Esto permite al frontend hacer CRUD sin crear un endpoint por cada tipo de nodo.

@router.patch("/{label}/{node_id}/property")
async def set_property(label, node_id, body):
  validate_label(label)  # lista blanca
  rows = await q(
    f"MATCH (n:{label}) WHERE n.id = $id "
    f"SET n[$prop] = $value RETURN n",
    id=node_id, prop=body.prop, value=body.value
  )
  return dict(rows[0]["n"])
→ Decorador: HTTP PATCH a esta ruta
→ Define función async con los parámetros de la URL
→ Verifica que el label esté en la lista autorizada
→ Llama a la función q() de database.py
→   Query Cypher: encuentra nodo y actualiza propiedad
 
→   Parámetros seguros (no concatenados al string)
 
→ Convierte el primer resultado a dict y devuelve
💡 Lista blanca de labels

ALLOWED_LABELS = {"Board","Thread","Post","User",...}. Sin esta validación, alguien podría enviar label="; DROP ALL DATA ;". La lista blanca bloquea cualquier label no autorizado.

Módulo 5 · Q1

Detectar brigading con IPs

Brigading: cuando varias cuentas coordinan desde las mismas IPs para inundar múltiples boards. Esta query lo detecta contando en cuántos boards distintos aparece cada IP.

MATCH (ip:IP)<-[:POSTED_FROM]-(p:Post)
    <-[:HAS_POST]-(t:Thread)
    <-[:HAS_THREAD]-(b:Board)
WITH ip,
     collect(DISTINCT b.slug) AS boards_hit,
     count(p) AS total_posts
WHERE size(boards_hit) > 2
RETURN ip.address, total_posts, boards_hit
→ Sigue la cadena: IP ← Post ← Thread ← Board
 
 
→ Agrupa por IP:
→   lista de boards únicos donde apareció
→   total de posts enviados
→ Solo IPs que aparecen en más de 2 boards
→ Devuelve: dirección IP, posts, y qué boards afectó
💡 ¿Por qué es fácil en un grafo?

La cadena IP → Post → Thread → Board ya existe como relaciones. Neo4j la recorre en milisegundos. En SQL necesitarías 3 JOINs + GROUP BY + HAVING para lograr lo mismo.

Módulo 5 · Q3 + Q7

Paths y PageRank

Q3 — Cadenas de citas

Cypher puede recorrer relaciones de profundidad variable con *1..6.

MATCH path = (root:Post)
  -[:QUOTES*1..6]->(leaf:Post)
WHERE NOT (root)<-[:QUOTES]-()
WITH path, length(path) AS depth
ORDER BY depth DESC
RETURN [n IN nodes(path) | n.id]

*1..6 = "sigue QUOTES entre 1 y 6 saltos". Encuentra la conversación más larga del foro.

Q7 — PageRank con GDS

GDS es el plugin de algoritmos de Neo4j. PageRank mide qué posts son más influyentes basándose en cuántas veces son citados.

CALL gds.graph.project(
  'post-graph', 'Post',
  {QUOTES: {orientation:'NATURAL'}}
)
CALL gds.pageRank.stream('post-graph')
YIELD nodeId, score
RETURN gds.util.asNode(nodeId).id, score

Primero proyecta el grafo en memoria, luego corre PageRank. Un score alto = post muy referenciado = influencia real.
